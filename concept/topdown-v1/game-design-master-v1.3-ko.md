## 1. 메인 기획서

# 슬라이드 퍼즐 게임 기획서 v1

- 작성일: 2026-03-05
- 버전: v1.3 (출시 범위 재정의 반영본)
- 범위: World 1 Normal/Hard 구현 + World 2~5 규칙 레퍼런스 보관
- 연결 문서: [슬라이드 퍼즐 게임 아트 가이드 v1](./art-guide-v1-ko.md)

## 1. 게임 개요

본 게임은 **슬라이드 이동 기반 경로 최적화 퍼즐**이다.  
플레이어는 상/하/좌/우 스와이프로 캐릭터를 이동시키며, 캐릭터는 입력한 방향으로 벽에 부딪힐 때까지 미끄러지듯 이동한다.

스테이지 목표는 다음 2단계다.

1. 키 3개를 모두 수집
2. 활성화된 포탈에 도달하여 클리어

핵심 재미는 실패 압박이 아닌, **최소 이동(슬라이드 횟수)으로 클리어하는 효율 도전**이다.

## 2. 코어 규칙

### 2.1 이동 규칙

- 입력: 모바일 스와이프(상/하/좌/우)만 지원
- 캐릭터는 입력 방향으로 연속 이동
- 벽 또는 이동 불가 타일을 만나면 정지
- 슬라이드 1회 = 이동 횟수 1로 카운트
- 슬라이드 시작 후 정지할 때까지 추가 입력은 무시(입력 잠금)

### 2.2 오브젝트 규칙

- 키는 총 3개
- 캐릭터가 슬라이드 중간에 키 타일을 지나가면 즉시 획득
- 포탈은 스테이지 입장 시 비활성 상태
- 키 3개 획득 시 포탈 활성화
- 비활성 포탈은 벽처럼 취급(통과/정지 불가)

### 2.3 클리어/실패 규칙

- 활성 포탈 타일에 캐릭터가 도달하면 스테이지 클리어
- 실패 조건 없음(언제든 클리어 가능)

## 3. 별(Star) 평가 시스템

- `M`: 해당 플레이(배치 기준)의 최소 이동 횟수
- `P`: 플레이어 실제 이동 횟수

평가 규칙:

- `P == M` => 별 3개
- `P <= M + 2` => 별 2개
- 그 외 => 별 1개

World 1 정책:

- 튜토리얼 난이도 기준으로 `M + 2` 규칙을 유지

## 4. 스테이지/월드 구조

### 4.1 기본 구조

- World 1은 `Normal(10 스테이지)` + `Hard(10 스테이지)`로 구성
- 모드별 스테이지 번호: `1 ~ 10`
- 각 모드는 직전 스테이지를 클리어해야 다음 스테이지 해금
- 클리어한 스테이지는 언제든 재도전 가능
- 스테이지 최종 별은 최고 기록만 유지

예시:

- Stage 1: 1회차 2별, 2회차 1별 -> 최종 2별 유지
- Stage 2: 1회차 1별, 2회차 3별 -> 최종 3별 갱신

### 4.2 모드/월드 해금 규칙 (출시 기준)

- World 1 Hard 오픈 조건: World 1 Normal `1~10` 전 스테이지 `3별` 달성
- World 2 Normal 오픈 조건: World 1 Normal `Stage 10` 클리어(별 1개 이상)
- 출시 버전에서는 World 2~5를 UI상 잠금 상태로 노출하고 플레이는 비활성화
- World 2~5의 실제 플레이/해금 로직은 후속 업데이트에서 적용

## 5. World 1 확정 정책 (출시 대상)

World 1은 튜토리얼 + 숙련 챌린지를 한 월드 내 2개 모드로 운영한다.

### 5.1 Normal 모드(튜토리얼)

- 맵 형태: 스테이지별 고정
- 시작 위치: 스테이지별 고정 1개
- 포탈 위치: 스테이지별 고정
- 키 3개 위치: 스테이지별 고정
- 키 위치 예외: 시작 타일, 포탈 타일에는 배치 금지
- Stage 1~3은 하단 중앙 시작 우선, Stage 4~10은 퍼즐 의도 기반 개별 시작점 사용

### 5.2 Hard 모드(출시 엔드 콘텐츠)

- Hard Stage 1~2: 키 순서 시스템(`1 -> 2 -> 3`)
- Hard Stage 3~5: 키 순서 + 포탈 생성 위치 랜덤
- Hard Stage 6~7: 키 순서 + 포탈 랜덤 + 일방통행 패널
- Hard Stage 8~10: 키 순서 + 포탈 랜덤 + 일방통행 + 워프 패널
- 포탈 랜덤 규칙: 키 3개 획득 시 후보 위치 중 1개를 랜덤 활성화
- 공정성 규칙: 랜덤 요소가 있는 스테이지는 해당 실행 배치 기준 최소 이동 `M`으로 별 판정

## 6. 저장 정책

- 저장 대상:
  - 모드별 스테이지 클리어 여부(`W1 Normal`, `W1 Hard`)
  - 모드별 스테이지 최고 별 기록
  - 해금 상태(`W1 Hard`, `W2 Normal`)
  - World 2~5 잠금 상태(출시 버전 고정 잠금)

## 1. 아트 목표

본 게임의 아트는 다음 3가지를 동시에 만족해야 한다.

1. 감성적이고 부드러운 분위기
2. 모바일에서 즉시 읽히는 퍼즐 가독성
3. 월드 기믹과 아트가 직관적으로 연결되는 테마성

## 2. 공통 시각 문법

월드가 바뀌어도 퍼즐 판독 기준은 동일해야 한다.

- 이동 가능 타일: 밝은 상면 + 낮은 대비 외곽선
- 벽/이동 불가 타일: 더 높은 높이감 + 강한 그림자
- 플레이어: 타일 대비가 가장 높은 실루엣
- 키: 작은 크기라도 보이도록 외곽 글로우 유지
- 포탈(비활성): 닫힌 문양/어두운 코어, 벽처럼 보이게 처리
- 포탈(활성): 회전 링 + 맥동 광원 + 진입 가능 느낌

상태 표현 규칙:

- 획득 전 키: 고유 색 + 미세 파티클
- 획득 후 키 UI: 상단 HUD 슬롯 점등
- 목표 달성(3개 획득): 포탈 활성화 VFX 즉시 재생
- 클리어: 별/배너 연출은 1초 내 핵심 정보 전달

## 3. 접근성 및 가독성 규칙

- 색상만으로 상태를 구분하지 않는다(아이콘/숫자/형태 동시 사용)
- 특히 World 2 키 순서 기믹은 `1,2,3` 숫자 표기를 기본값으로 사용
- 버튼 터치 영역 최소 44px
- 본문 UI 대비 비율 권장: 4.5:1 이상
- 배경 이펙트는 플레이 타일 가독성을 침범하지 않도록 채도/밝기 제한

## 4. 월드별 컨셉 가이드

## 4.1 World 1 - Frozen Path (Basic)

- 키워드: 차분함, 학습, 선명한 판독
- 지형 소재: 얼음, 눈, 옅은 서리 결정
- 주조색: 아이스 블루 계열
- 기믹 연결: 기본 규칙 학습, 타일 경계 명확성 최우선
- 연출 톤: 부드러운 냉광, 과한 입자 사용 금지

## 4.2 World 2 - Blazing Depths (Key Order)

- 키워드: 긴장감, 순서, 리스크 인지
- 지형 소재: 용암 균열, 화산암, 제단
- 주조색: 오렌지/적색 + 짙은 흑갈
- 기믹 연결: 키 순서 `1 -> 2 -> 3`을 아트로 즉시 인지
- 연출 톤: 균열 발광/열기 왜곡은 가능하나 타일 경계는 유지
- 접근성: 빨강/초록만 사용 금지, 숫자/문양 동시 표기 필수

## 4.3 World 3 - Starlit Garden (Dynamic Exit)

- 키워드: 신비, 발견, 반응형 공간
- 지형 소재: 부유 섬, 꽃/수풀, 별빛 파편
- 주조색: 보라/청보라 + 포인트 네온
- 기믹 연결: 마지막 키 획득 시 출구 생성 연출이 핵심
- 연출 톤: 포탈 생성 순간의 공간 왜곡/광원 전환 강조

## 4.4 World 4 - Ancient Ruins (One-Way)

- 키워드: 규칙성, 고대 장치, 방향성
- 지형 소재: 사암 유적, 각인된 화살표 판석
- 주조색: 샌드 브라운/골드
- 기믹 연결: 일방 통행 방향을 타일 문양으로 항상 표시
- 연출 톤: UI 없이도 방향 판독 가능해야 함

## 4.5 World 5 - Void Nexus (Teleport)

- 키워드: 초현실, 단절, 연결
- 지형 소재: 부유 암석, 균열 에너지, 보이드 웅덩이
- 주조색: 딥 블루/퍼플 + 시안 네온
- 기믹 연결: 텔레포트 쌍의 시각적 페어링 강조
- 연출 톤: 입출구 색/문양을 동일하게 맞춰 학습 부담 최소화

## 2. 핵심 규칙

## 2.1 키 순서 규칙

## 2.2 잘못된 순서 처리(확정)

- 잘못된 키 타일을 지나가거나 도착해도 이동 자체는 정상 진행

- 즉시 경고 피드백 표시
  - 시각: 짧은 붉은 경고 링/흔들림
  - 오디오: 짧은 "not-allowed" 톤
  - UI: `다음 키: N` 강조 점멸 1회

이 방식은 "No fail" 철학을 유지하면서도 순서 학습을 강제한다.

## 2.3 포탈 규칙

- 포탈은 입장 시 비활성

운영 메모:

- World 2는 상태 기믹으로 체감 난도가 상승하므로, 스테이지 설계 시 `M` 산정이 과도하게 빡빡해지지 않도록 솔버 검증을 필수 적용

  - [슬라이드 퍼즐 게임 아트 가이드 v1](./art-guide-v1-ko.md)
  - [World 2 기획서 v1 - Blazing Depths](./world2-design-v1-ko.md)

## 1. World 3 목표

World 3는 "출구가 고정된 퍼즐"에서 벗어나, **키 수집 경로가 출구 위치를 결정**하는 구조를 학습시키는 월드다.

- World 2 핵심: 상태(키 순서) 기반 퍼즐
- World 3 핵심: 결과(마지막 키 위치) 기반 퍼즐

플레이어는 "어디로 갈지"뿐 아니라 "어떤 키를 마지막으로 먹을지"까지 설계해야 한다.

## 2. 핵심 규칙

## 2.1 키 규칙

- 키는 총 3개
- World 3에서는 키 순서 제한 없음(어떤 순서로 먹어도 됨)
- 슬라이드 중간 통과 시 즉시 획득

## 2.2 출구(포탈) 규칙

- 스테이지 시작 시 포탈은 **미생성** 상태(보이지 않음)
- 키 3개를 모두 획득하는 순간 포탈 후보 중 1개가 활성화됨
- 활성 포탈 도달 시 스테이지 클리어

World 3 예외:

- 기존 "비활성 포탈은 벽처럼 처리" 규칙 대신, World 3는 포탈이 생성되기 전에는 포탈 타일 자체가 존재하지 않는다.

## 2.3 동적 출구 선택 알고리즘(확정안)

포탈 후보를 `candidates[]`라 할 때:

전제:

- 키 3개 완성 시점이 슬라이드 중간이어도, 포탈 선택/생성은 **해당 슬라이드 정지 후** 실행

1. `lastKeyTile` 계산
   - 세 번째 키를 획득한 타일을 `lastKeyTile`로 기록
   - 한 번의 슬라이드에서 남은 키 2개 이상을 연속 획득해 3개가 완성되면, 슬라이드 경로에서 **마지막으로 획득된 키 타일**을 사용
2. `activationTile` 계산
   - 3개 완성 직후 현재 슬라이드가 끝났을 때의 플레이어 정지 타일
3. `activationTile`에서 도달 가능한 후보만 필터링
4. 필터링된 후보에 대해 `lastKeyTile -> candidate` 슬라이드 최단 이동 수를 계산
5. 최단 이동 수가 가장 작은 후보를 선택
6. 동점이면 `priority`가 가장 낮은 후보를 선택
7. 선택된 후보 1개만 활성 포탈로 생성

Fail Fast:

- 도달 가능한 후보가 0개인 스테이지 데이터는 잘못된 데이터로 간주하고 빌드 검증에서 에러 처리

## 3. 별 평가 정책

기본 별 산식은 동일:

- `P == M` => 3별
- `P <= M + 2` => 2별
- 그 외 => 1별

World 3 주의:

- `M` 계산 시 상태공간에 `키 집합 + lastKeyTile + activationTile + 활성 포탈 후보`가 포함되어야 함

## 4. 난이도 곡선 (10 스테이지)

## 4.1 목표 난이도 구조

- Stage 1~3: 동적 출구 개념 학습(후보 2개)
- Stage 4~6: 마지막 키 선택 유도(후보 3개)

## 4.2 Stage별 목표 최소 이동(`M`) 권장값

- Stage 1: `M=5`
- Stage 2: `M=6`
- Stage 3: `M=6`
- Stage 4: `M=7`

## 5. 데이터 설계 반영 항목

## 7.1 설계 홀(리스크)와 대응

1. `lastKeyTile` 정의 모호성
   - 리스크: 한 슬라이드에서 키 2개 이상 획득 시 결과 불일치
   - 대응: "3개 완성 순간, 경로상 마지막으로 획득된 키 타일"로 고정

2. 후보 거리 동점 처리 부재
   - 리스크: 같은 입력에서도 포탈 위치가 달라지는 비결정성
   - 대응: `priority` 필드 기반 타이브레이크 고정

3. 가장 가까운 후보가 도달 불가능한 경우
   - 리스크: 규칙상 가까운 후보를 택했는데 실제 클리어 불가
   - 대응: `activationTile`에서 도달 가능한 후보만 먼저 필터링

4. 후보 전부 도달 불가
   - 리스크: 노실패 철학 위반(실질 소프트락)
   - 대응: 스테이지 데이터 검증 단계에서 Fail Fast 에러

5. `M` 계산 누락
   - 리스크: 별 평가가 체감 난도와 어긋남
   - 대응: 솔버 상태에 `키 집합 + lastKeyTile + activationTile + 활성 포탈 후보` 포함

6. 동점 타이브레이크 체감 불투명
   - 리스크: 플레이어가 "왜 이 포탈이 열렸는지" 납득하기 어려움
   - 대응: 튜토리얼 텍스트/팁에서 "동점 시 우선순위 규칙"을 명시

1. 3번째 키 획득과 동시에 활성 포탈 타일에 플레이어가 서 있는 경우

- World 3 핵심: 결과(출구 생성 위치) 제어
- World 4 핵심: 경로의 방향 제약 제어

플레이어는 같은 지점이라도 접근 방향이 다르면 결과가 달라진다는 점을 학습한다.

## 2. 핵심 규칙

## 2.1 일방통행 패널 규칙 (확정)

패널은 화살표 방향(`up/down/left/right`)을 가진다.

- 패널은 **진입 방향**을 검사한다.
- 이동 방향이 패널 화살표와 같을 때만 패널 진입 가능
- 불일치하면 패널은 벽처럼 취급되어 이전 타일에서 정지
- 패널에 진입한 뒤에는 일반 바닥처럼 취급(퇴장 방향 제한 없음)

확정 이유:

- "No Fail" 철학에 맞게 소프트락 위험이 낮다.
- 룰 인지는 쉽고, 스테이지 설계 난이도는 충분히 확보 가능하다.
- World 5에서 워프와 혼합해도 규칙 충돌이 상대적으로 적다.

## 2.2 키/포탈 규칙

- 키 3개 획득 후 포탈 활성화(기본 규칙 유지)
- 비활성 포탈은 벽처럼 취급(기본 규칙 유지)
- 키는 시작/포탈 타일 위 배치 금지

## 2.3 이동/카운트 규칙

- 슬라이드 중 입력 잠금 유지
- 유효 이동(최소 1타일 이동)만 이동 횟수 +1
- 첫 스텝부터 벽/패널 차단이면 이동 횟수 증가 없음

## 3. 일방통행 판정 알고리즘 (제안안)

슬라이드 시 다음 타일(`next`)을 검사할 때:

1. `next`가 벽이면 정지
2. `next`가 일방통행 패널이면
   - `moveDirection == panel.direction`인 경우 진입 허용
   - 아니면 현재 타일에서 정지
3. 그 외 타일은 일반 규칙대로 진행

## 4. 난이도 곡선 (10 스테이지)

## 4.1 목표 난이도 구조

- Stage 1~3: 패널 읽기 학습(단일 방향 게이트)
- Stage 4~6: 우회 경로 강제/분기 증가
- Stage 7~8: 다중 패널 연쇄, 역방향 유도 함정
- Stage 9~10: 패널 네트워크 종합 퍼즐

## 4.2 Stage별 목표 최소 이동(`M`) 권장값

- Stage 1: `M=6`
- Stage 2: `M=7`
- Stage 3: `M=7`
- Stage 4: `M=8`
- Stage 5: `M=9`
- Stage 6: `M=9`
- Stage 7: `M=10`
- Stage 8: `M=10`
- Stage 9: `M=11`
- Stage 10: `M=12`

## 5. 데이터 설계 반영 항목

World 4 스테이지 데이터 필드:

- `worldMechanic`: `"one-way-panel"`
- `oneWayPanels[]`
  - `id`
  - `position`
  - `direction`: `"up" | "down" | "left" | "right"`
  - `mode`: `"entry-gate"` (v1 고정)

검증 규칙:

- 패널은 벽/시작/키/포탈과 좌표 중복 금지
- 패널 방향 표식은 타일 아트와 데이터가 일치해야 함
- 모든 키 완료 상태에서 포탈까지 도달 가능한 경로가 1개 이상 존재해야 함

## 6. UX/연출 가이드

- 패널 화살표는 배경과 분리되도록 고대 문양 + 글로우로 표현
- 접근 불가 방향에서 충돌 시:
  - 짧은 "튕김" VFX
  - 짧은 금속성 차단 SFX
- HUD 도움말(초반 1~2스테이지): `화살표 방향으로만 통과 가능`

## 7. 예상 홀/예외사항 검토

## 7.1 설계 홀(리스크)와 대응

1. 패널 방향 판독 실패
   - 리스크: 퍼즐 난도가 아닌 시인성 문제로 실패 체감
   - 대응: 방향 아이콘 크기/대비 기준을 아트 가이드에 고정

2. 이동 카운트 악용
   - 리스크: 차단 입력이 카운트되면 난이도 왜곡
   - 대응: 유효 이동만 카운트(0타일 이동 미카운트)

3. 패널 네트워크로 인한 실질 소프트락
   - 리스크: 특정 상태에서 포탈 도달 경로 상실
   - 대응: 스테이지 검증에서 "모든 키 획득 후 포탈 도달 가능" 검사

4. 패널 위 키 배치의 판독 혼란
   - 리스크: 키와 화살표가 겹쳐 가독성 저하
   - 대응: v1에서는 **패널 타일과 키의 동일 좌표 배치 금지**

5. 규칙 확장 충돌
   - 리스크: 추후 "퇴장 제한" 모드 추가 시 기존 스테이지 깨짐
   - 대응: `mode` 필드로 명시적 버전 관리(`entry-gate` 고정)

## 7.2 런타임 예외 시나리오

1. 첫 스텝이 패널 차단인 경우
   - 처리: 캐릭터 위치 유지, 이동 횟수 증가 없음

2. 패널 차단과 키 획득이 같은 프레임에 충돌하는 경우
   - 처리: 패널 차단 우선, 키 미획득

3. 포탈 활성화 연출 중 패널 충돌 입력
   - 처리: 입력 잠금 유지, 연출 후 입력 재개

## 8. QA 체크리스트

1. 허용 방향에서만 패널 진입이 되는가
2. 차단 입력이 이동 횟수에 반영되지 않는가
3. 패널 차단 시 키/포탈 판정이 잘못 발생하지 않는가
4. 모든 스테이지가 소프트락 없이 클리어 가능한가
5. `M`/별 산정이 의도 난이도와 일치하는가

## 9. 구현 우선순위

1. 일방통행 패널 판정 로직 구현(`entry-gate`)
2. 이동 카운트 정책(유효 이동만 카운트) 반영
3. World 4 스테이지 데이터 10개 작성
4. 스테이지 검증기(도달성/중복/시각표식 일치) 추가
5. 패널 충돌 피드백(VFX/SFX) 연결

## 10. 확정값

1. World 4 패널 모드는 `entry-gate` 고정
2. World 4에서는 패널 타일과 키의 동일 좌표 배치 금지

World 5는 맵의 물리적 거리보다 **공간 연결 관계(그래프 구조)**를 이해하는 퍼즐로 확장한다.

## 2. 핵심 규칙

## 2.1 워프 패널 규칙

- 워프 패널은 쌍(`pairId`)으로 존재
- 슬라이드 중 워프 패널에 진입하면 즉시 짝 패널로 이동
- 텔레포트 자체로 이동 횟수는 증가하지 않음(해당 스와이프 1회 유지)
- 워프 후 이동 방향은 유지

## 2.2 트리거 규칙 (루프 방지 확정안)

- "인접 타일에서 워프 타일로 진입"할 때만 워프 트리거
- 워프로 도착한 타일은 **즉시 재트리거하지 않음**
- 같은 스와이프에서 동일 `pairId` 워프를 다시 진입하면 재트리거 금지
- 재진입 시 해당 워프는 **일반 바닥처럼 통과**(차단하지 않음)

선택 이유:

- 무한 워프 루프를 구조적으로 차단
- 입력 1회당 결과가 항상 결정론적으로 고정

## 2.3 키/포탈 규칙

- 키 3개 획득 후 포탈 활성화(기본 규칙 유지)
- 비활성 포탈은 벽처럼 취급(기본 규칙 유지)
- 키는 시작/포탈 타일 위 배치 금지

## 2.4 키 획득 타이밍 규칙

- 워프 진입 타일에 키가 있으면, 진입 즉시 획득 후 텔레포트
- 워프 도착 타일에 키가 있으면, 도착 즉시 획득
- 한 스와이프에서 키 2개 이상 획득 가능
- World 5에서는 워프 타일 위 키 배치를 허용

## 2.5 혼합 기믹 규칙 (World 5 확장)

- World 5에서는 일방통행 패널(World 4 규칙)과 워프 패널 혼합 배치를 허용
- 일방통행 패널은 `entry-gate` 모드만 사용
- 워프 타일과 일방통행 패널 타일의 **동일 좌표 중첩은 금지**

## 3. 워프 이동 알고리즘 (제안안)

슬라이드 시 다음 타일(`next`)을 검사할 때:

1. `next`가 벽이면 정지
2. `next`가 일방통행 패널이고 `moveDirection != panel.direction`이면 현재 타일에서 정지
3. `next`가 워프 타일이고 `pairId(next)`가 이번 스와이프에서 미사용이면
   - `next` 진입 처리
   - 진입 타일 키 획득 판정
   - `pairId` 목적지로 즉시 이동
   - 목적지 키 획득 판정
   - 사용 워프 집합에 `pairId(next)` 추가
   - 같은 이동 방향으로 계속 진행
4. `next`가 워프 타일이지만 `pairId(next)`가 이번 스와이프 사용 이력이 있으면
   - 일반 타일처럼 통과(재트리거 금지)
5. 그 외 타일은 일반 규칙대로 진행

## 7. 예상 홀/예외사항 검토

## 7.1 설계 홀(리스크)와 대응

1. 무한 워프 루프
   - 리스크: 프레임 고정/입력 불능
   - 대응: 스와이프 단위 `pairId` 재트리거 금지 + 사용 이력 집합 적용

2. 워프 후 즉시 역워프
   - 리스크: 플레이어가 이동 결과를 예측하기 어려움
   - 대응: "워프 쌍은 스와이프당 1회" 규칙으로 역워프 차단

3. 워프 타일 위 오브젝트 중첩
   - 리스크: 키/포탈 판정 순서 버그
   - 대응: 키 획득 타이밍(진입->도착)을 규칙으로 고정, 포탈 중첩 금지

4. 이동 횟수 산정 불일치
   - 리스크: 텔레포트 횟수만큼 이동 수가 늘어나 별 평가 왜곡
   - 대응: 이동 횟수는 입력(스와이프) 단위만 카운트

5. 솔버 비결정성
   - 리스크: 실제 플레이와 `M` 결과 불일치
   - 대응: 솔버 시뮬레이터에 `pairId` 단위 재트리거 금지 규칙 동일 반영

6. 일방통행+워프 혼합 충돌
   - 리스크: 판정 우선순위가 불명확하면 결과 불일치 발생
   - 대응: 동일 좌표 중첩 금지 + 타일 진입 판정(일방통행) 후 워프 판정 순서 고정

## 7.2 런타임 예외 시나리오

1. 워프 목적지 타일이 데이터 오류로 막힌 경우
   - 처리: 로딩 시 검증 에러 throw, 스테이지 진입 차단

2. 워프 도착 직후 포탈 활성화가 동시에 발생하는 경우
   - 처리: 상태 업데이트(키/포탈) 후 즉시 클리어 판정

3. 포탈 활성화 연출 중 워프 진입 입력
   - 처리: 입력 잠금 유지, 연출 종료 후 입력 재개

## 8. QA 체크리스트

1. 워프 1회 사용이 이동 횟수 1회로 유지되는가
2. 같은 스와이프 내 재진입 워프가 재트리거되지 않는가
3. 워프 진입/도착 키 획득 순서가 규칙대로 동작하는가
4. 워프+일방통행 혼합 스테이지가 소프트락/무한루프 없이 클리어 가능한가
5. `M`/별 산정이 실제 플레이와 일치하는가

## 9. 구현 우선순위

1. 워프 트리거/재트리거 금지 로직 구현
